# 技术分析：Cover Generation Server

我把这个项目定位为一个“可复现、可版本化的 SVG 封面渲染服务”。它不追求通用的排版/设计编辑器能力，而是追求：输入字段足够少、输出样式稳定、渲染性能高、可控可演进。

本文从技术与架构角度解释：我用到了哪些技术、模块如何协作、为什么这样设计，以及未来如何扩展版本（v2/v3/v4/v5/v6/v7）。

## 1. 整体架构（分层职责）

我将系统拆成三层，尽量降低耦合：

1) HTTP 层（`src/server.js`）
- 负责路由、请求体解析、错误处理、响应头。
- 负责安全边界：限制 JSON body 最大 512KB，避免意外大 payload。
- 不参与“渲染细节”，只调用渲染入口并返回结果。

2) 参数归一化层（`src/coverGenerator/index.js`）
- 负责把 query/body 合并成 `options`。
- 负责输入裁剪（title/subtitle/author 长度上限）与数值范围（width/height clamp）。
- 负责 `seed` 归一化：把 seed 变成稳定 uint32，从而保证可复现随机。
- 这是对外 API 的“防火墙”：模板层只接收干净、有限、稳定的输入。

3) 渲染层（`src/coverGenerator/exporter.js` + `shapeEngine.js` + `typographyEngine.js`）
- `exporter.js`：模板注册与调度（当前支持 `v1`/`v2`/`v3`/`v4`/`v5`/`v6`/`v7`）。
- `config.js`：随机封面默认作者与头像列表配置。
- `templates/v1.js`：`v1` 模板实现，输出完整 SVG 字符串。
- `shapeEngine.js`：复用组件（当前最重要的是头像渲染）。
- `typographyEngine.js`：文本安全与基础排版工具。
- `utils.js`：seed 与可复现随机相关工具。

这样的好处：
- 我可以在不动 HTTP 层的情况下，新增模板版本或调整渲染策略。
- 我可以在不动模板层的情况下，增强输入校验/参数定义。
- 我可以通过版本号对外隔离破坏性变化。

## 2. 为什么选 SVG

我选择 SVG 的原因：
- 文本仍然是“文本”：清晰、可搜索、可缩放，不像位图那样需要额外字体渲染和抗锯齿策略。
- 输出是单个字符串：便于服务端生成、便于缓存、便于在构建时落盘。
- 可以直接被浏览器或构建工具消费：例如把 SVG 交给后续的 PNG 转换流程。

## 3. 可复现随机（seedable）

可复现是这个项目的核心能力之一。

### 3.1 seed 的归一化
入口在 `src/coverGenerator/utils.js`：
- `normalizeSeed(seedInput, fallbackKey)`：
  - number => uint32
  - string => hash => uint32
  - 未提供 => hash(fallbackKey)

默认 fallbackKey 通常包含 `title/author/template`，所以即使不传 seed，也能“相对稳定”（同输入同输出）。

### 3.2 PRNG
同样在 `utils.js`：
- `createRng(seed)`：返回一个 PRNG 函数（Mulberry32）。

模板层只使用这个 rng 来做颜色/渐变角度等随机决策，从而保证：
- 同 seed => 同颜色选择、同渐变角度、同卡片色选择

## 4. 模板 v1 的渲染设计

当前的 v1 是“暖色背景 + 浅色卡片”的视觉体系，包含：
- 背景：暖色系纯色或暖色系渐变
- 卡片：随机浅色（或由 `accent` 固定）
- 标题：居中、粗体、自动换行/缩放
- footer：左下角头像（emoji/URL/占位符），右下角作者文本

### 4.1 背景策略
在 `src/coverGenerator/templates/v1.js`：
- `background=solid`：纯色
- `background=gradient`：渐变（暖色系随机挑两色 + 随机角度）
- `background=auto`：在纯色/渐变之间按概率选择
- 如果传了 `color`：我把它视为“背景主色锁定”，会直接使用纯色背景（不再随机渐变），以便你在外部明确控制视觉。

### 4.2 卡片底色策略
- 如果传了 `accent`：卡片底色固定为 `accent`
- 否则：从一组浅色里随机选择（偏白、偏浅暖、偏浅冷都可以，但整体保持轻盈）

### 4.3 头像策略（占位符 + emoji 优先）
在 `src/coverGenerator/shapeEngine.js` 与 `exporter.js` 的 footer 渲染逻辑中：
- `avatarEmoji` 优先于 `avatarUrl`：避免同时存在时出现不确定依赖（头像 URL 可能不可用）。
- 两者都不传：渲染一个默认占位图形，让布局稳定。
- `avatarUrl` 只是 `<image href="...">`：服务端不下载，不做代理，也不引入网络 IO。

### 4.4 标题“动态换行 + 动态缩放”
这是我为了避免溢出卡片而做的关键设计：

- 我先根据卡片可用宽度进行换行；
- 如果换行后总高度仍超过可用高度，我会逐步降低 font-size 并重新换行；
- 直到文本高度满足约束或达到最小字号阈值。

混合中英文/emoji 时，简单按空格分词会导致估算偏差，所以我在 v1 的实现中采用：
- ASCII 片段按“单词块”合并；
- CJK/emoji 按“字符级”参与换行；
- 并用单位宽度估算来替代真实字体测量（服务端无字体度量环境，且我不希望引入额外依赖）。

这不是排版引擎级别的精确度，但足够稳定、性能好，并且在封面场景可接受。

## 5. 安全与边界

我把安全边界放在 HTTP 层和参数层：
- `src/server.js`：限制 body 最大 512KB，异常返回 400/413。
- `src/coverGenerator/index.js`：裁剪长度、clamp 数值范围，避免被超长文本拖垮 SVG。
- `escapeXml`：防止用户输入破坏 SVG/XML 结构。

我刻意不做：
- 服务端下载头像（避免 SSRF、超时、缓存复杂度、白名单维护）
- 支持任意复杂的背景 pattern（容易造成 SVG 体积暴涨、渲染性能不可控）

## 6. 版本化策略：v1 / v2 / v3 / v4 / v5 / v6 / v7

我把版本号当成“对外稳定契约”：
- `v1`：当前这套视觉体系的稳定实现。
- `v2`/`v3`：当我需要大改版式、默认风格、排版策略且不希望影响历史封面时，再新增版本。

当前仓库已实现：
- `v1`：居中卡片风格（暖色背景/可渐变 + 浅色卡片）
- `v2`：左侧大头像 + 右侧卡片文案（更适合“人物标识 + 大标题信息块”）
- `v3`：纯背景大标题（左上角头像 + 左对齐大标题 + 左下作者）
- `v4`：电路纹理背景 + 居中图标/标题（更适合技术风格封面）
- `v5`：暖色渐变网格 + 顶部标签 + 居中标题/单图标（更适合产品发布风格封面）
- `v6`：极简浅色背景 + 居中图标/标题（更适合产品/目录类封面）
- `v7`：粉色渐变背景 + 左图标右文案（更适合品牌/产品介绍）

### 6.1 什么情况下升版本
- 布局体系变化：例如从居中卡片变成左右分栏、顶部横条、卡片消失等
- 默认视觉变化：例如从暖色系变成冷色系/黑金/极简黑白等
- 排版策略变化：例如标题逻辑从“缩放适配”改为“固定字号 + 更多行”等

### 6.2 什么情况下加参数（仍在 v1 内）
- 同一体系内的可选项：例如 `background` 的选择、阴影强度、圆角大小、对齐方式
- 参数不应改变 v1 的“默认辨识度”，否则 v1 就不再是稳定契约

## 7. 如何扩展到 v2/v3/v4/v5/v6/v7（建议做法）

如果我要新增 v2，我会做：
- 在 `exporter.js` 增加 `renderTemplateV2(options)` 并加入 `templates.v2`
- 在 `src/server.js` 放开路由匹配（`/cover/svg/v2`）
- 在 `README.md` 增加 v2 的参数差异说明（避免“参数看似支持但其实不生效”）

并且保持：
- v1 的输出不变（除非明确宣布破坏性变化）
- v2 的默认值自洽、可复现（seed 仍然是缓存键的一部分）
